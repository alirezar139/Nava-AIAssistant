# استاندارد مدیریت خطا

## ساختار پاسخ API

همه خطاهای API باید ساختار ثابت زیر را داشته باشند:

```json
{
  "code": "FAQ_NOT_FOUND",
  "message": "FAQ موردنظر پیدا نشد.",
  "traceId": "a1b2c3d4"
}
```

- `code`: شناسه پایدار و قابل استفاده در کد
- `message`: پیام فارسی و قابل فهم برای کاربر
- `traceId`: شناسه پیگیری برای اتصال پیام کاربر به لاگ سرور

## اصول پیام کاربر

- پیام باید بگوید چه اتفاقی افتاده و کاربر چه اقدامی انجام دهد.
- جزئیات فنی، Stack Trace و اطلاعات امنیتی نباید نمایش داده شوند.
- خطای شبکه، اعتبارسنجی، دسترسی، نشست و خطای داخلی پیام‌های متفاوت دارند.
- عملیات محلی مانند فایل Excel نیز باید علت قابل اصلاح ارائه کند.
- خطاهای داخلی باید شناسه پیگیری داشته باشند.

## وضعیت‌های اصلی

| وضعیت | عنوان کاربر | اقدام پیشنهادی |
| --- | --- | --- |
| `0` | ارتباط با سرور برقرار نیست | بررسی API و شبکه |
| `400` | اطلاعات نیاز به اصلاح دارد | اصلاح ورودی‌ها |
| `401` | ورود نامعتبر است | ورود مجدد |
| `403` | دسترسی مجاز نیست | تماس با مدیر سامانه |
| `404` | اطلاعات پیدا نشد | تازه‌سازی فهرست |
| `409` | تداخل اطلاعات | دریافت نسخه جدید |
| `413` | حجم اطلاعات زیاد است | کاهش حجم فایل |
| `429` | تعداد درخواست زیاد است | تلاش مجدد پس از مکث |
| `5xx` | خطای داخلی سامانه | تلاش مجدد و ارائه شناسه پیگیری |
