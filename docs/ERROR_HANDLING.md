# مدیریت خطا

## هدف

خطاها باید برای UI قابل پیش‌بینی، برای پشتیبانی قابل پیگیری و برای کاربر امن
باشند. سامانه نباید stack trace، secret یا جزئیات داخلی integration را در پیام
قابل نمایش نشان دهد.

## قالب خطای API

خطاهای جدید API باید تا حد امکان این ساختار را داشته باشند:

```json
{
  "code": "FAQ_NOT_FOUND",
  "message": "FAQ موردنظر پیدا نشد.",
  "traceId": "a1b2c3d4"
}
```

| فیلد      | کاربرد                                    |
| --------- | ----------------------------------------- |
| `code`    | شناسه پایدار و قابل استفاده در کد.        |
| `message` | پیام فارسی قابل نمایش به کاربر.           |
| `traceId` | شناسه اتصال پیام کاربر به لاگ و پشتیبانی. |

## نگاشت وضعیت HTTP

| وضعیت | معنی                      | اقدام کاربر                             |
| ----- | ------------------------- | --------------------------------------- |
| `0`   | API یا شبکه در دسترس نیست | بررسی اجرای API و شبکه.                 |
| `400` | داده درخواست نامعتبر است  | اصلاح ورودی و تلاش مجدد.                |
| `401` | نشست نامعتبر یا منقضی است | ورود دوباره.                            |
| `403` | نقش کاربر مجاز نیست       | تماس با مدیر سامانه.                    |
| `404` | داده پیدا نشد             | تازه‌سازی یا بررسی شناسه.               |
| `409` | تداخل داده                | دریافت نسخه جدید و اعمال دوباره.        |
| `413` | حجم payload زیاد است      | کاهش حجم فایل یا تصویر.                 |
| `429` | درخواست بیش از حد         | تلاش دوباره بعد از مکث.                 |
| `5xx` | خطای داخلی یا integration | تلاش مجدد و ارسال trace id به پشتیبانی. |

## قواعد پیام UI

- پیام بگوید چه اتفاقی افتاده است.
- در صورت امکان اقدام بعدی را مشخص کند.
- برای layoutهای فشرده کوتاه و خوانا باشد.
- stack trace یا پاسخ خام سامانه مقصد نمایش داده نشود.
- خطای شبکه، validation، auth، permission، not found و internal پیام جدا داشته
  باشند.
- خطاهای پردازش فایل Excel یا Word باید علت قابل اصلاح را مشخص کنند.

## خطاهای تیکت

ثبت تیکت دو لایه دارد:

- ایجاد پرونده تشخیصی داخلی.
- ارسال اختیاری به سهند/Jira.

شکست ارسال خارجی نباید اطلاعات پرونده داخلی را حذف کند. UI باید وضعیت flow را
ناموفق نشان دهد و اطلاعات کافی برای بررسی یا تلاش مجدد نگه دارد.

## چک‌لیست بازبینی

- route جدید قبل از نوشتن داده، input را validate کند.
- خطای جدید `code` پایدار داشته باشد.
- UI برای خطای ناشناخته fallback داشته باشد.
- secret یا stack trace به client برنگردد.
- شکست integration تیکت، پرونده داخلی را حفظ کند.
