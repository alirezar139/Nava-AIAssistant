# معماری

## هدف

این سند معماری فعلی راهیار را توضیح می‌دهد: مرزهای سیستم، اجزای
اجرایی، مالکیت داده و جریان اصلی پشتیبانی.

## نمای کلی سیستم

```text
کاربر / مدیر
    |
    v
Angular SPA
    |
    v
Express API
    |
    +--> ذخیره‌سازی پیش‌فرض LowDB JSON
    |
    +--> اتصال اختیاری ArangoDB برای داده‌های عملیاتی و graph درختواره
    |
    +--> اتصال اختیاری به سهند/Jira
```

فرانت‌اند یک برنامه تک‌صفحه‌ای Angular است. بک‌اند یک API با Express است که در
یک پردازش Node جدا اجرا می‌شود. داده‌های عملیاتی در اجرای پیش‌فرض با فایل JSON
محلی ذخیره می‌شوند. اگر `DB_PROVIDER=arango` تنظیم شود، API اتصال ArangoDB را
بررسی می‌کند، دیتابیس و collectionهای لازم را می‌سازد و داده‌های عملیاتی، تنظیمات
و درختواره عیب‌یابی را از ArangoDB می‌خواند و در همان‌جا ذخیره می‌کند.

در اجرای توسعه، Angular dev server روی پورت `4200` و API روی پورت `3000` اجرا
می‌شوند. فرانت‌اند درخواست‌های `/api` را از طریق `proxy.conf.json` به API منتقل
می‌کند. در اجرای build شده، Express خروجی `dist/nava-ai-assistant` را هم سرو
می‌کند تا کل وب‌اپ با یک پردازش Node روی ویندوز اجرا شود.

فایل‌های PWA شامل `manifest.webmanifest` و `nava-service-worker.js` در build
تولیدی به خروجی اضافه می‌شوند. آیکن نصب بعد از دریافت فایل نهایی برند دوباره به
manifest اضافه می‌شود.

## معماری فرانت‌اند

```text
src/app/
|-- core/       مدل‌ها و سرویس‌های مشترک
|-- features/   دامنه‌های route مثل auth، admin و assistant
|-- shared/     کامپوننت‌های نمایشی قابل استفاده مجدد
|-- app.config.ts
|-- app.routes.ts
`-- app.component.ts
```

قواعد:

- `core` نباید از featureها import کند.
- featureها می‌توانند از `core` و `shared` استفاده کنند.
- کامپوننت‌های `shared` نباید منطق workflow دامنه داشته باشند.
- صفحه‌ها lazy load می‌شوند.
- TypeScript و template checking باید strict بمانند.

## معماری بک‌اند

```text
server/src/
|-- auth/             ورود، CAPTCHA، JWT و middleware
|-- common/           ابزارهای مشترک API
|-- config/           تنظیمات زمان اجرا
|-- conversations/    ثبت گفتگو و گزارش مدیریتی
|-- database/         راه‌اندازی LowDB، repositoryها و اتصال ArangoDB
|-- diagnostics/      پرونده تشخیصی و تحلیل اولیه
|-- faqs/             CRUD و import برای FAQ
|-- sahand/           اتصال اختیاری به سامانه تیکت
|-- services/         کاتالوگ سرویس‌های قابل تعریف
|-- settings/         تنظیمات قابل تغییر سامانه
|-- troubleshooting-tree/ درختواره عیب‌یابی و graph adapter
`-- main.ts           راه‌انداز Express
```

هر ماژول بک‌اند اعتبارسنجی درخواست، کنترل دسترسی، ذخیره‌سازی و شکل پاسخ همان
دامنه را مدیریت می‌کند.

`main.ts` بعد از ثبت routeهای `/api` بررسی می‌کند که خروجی Angular ساخته شده
باشد. اگر `dist/nava-ai-assistant/index.html` وجود داشته باشد، فایل‌های static
را سرو می‌کند و routeهای غیر API را به `index.html` برمی‌گرداند تا refresh صفحه
یا باز کردن مستقیم routeهای Angular کار کند.

service worker فقط خارج از Angular dev server ثبت می‌شود تا cache مرورگر در
توسعه روزمره مزاحم تغییرات UI نشود. مسیرهای `/api` توسط service worker cache
نمی‌شوند و همیشه از شبکه خوانده می‌شوند.

## احراز هویت و دسترسی

- ورود با نام کاربری، رمز عبور، CAPTCHA token و پاسخ CAPTCHA انجام می‌شود.
- رمزها با `bcryptjs` بررسی می‌شوند.
- ورود موفق JWT برمی‌گرداند.
- مسیرهای محافظت‌شده از `requireAuth` استفاده می‌کنند.
- مسیرهای مخصوص مدیر از `requireAuth(['admin'])` استفاده می‌کنند.
- کاربر احراز هویت‌شده می‌تواند گفتگو و پرونده تشخیصی ثبت کند.

## جریان اصلی پشتیبانی

```text
نمایش گزینه‌های درختواره
    |
    v
انتخاب مسیر توسط کاربر
    |
    v
جست‌وجو در FAQهای تاییدشده
    |
    +--> پاسخ قابل اتکا پیدا شد
    |       |
    |       v
    |   نمایش پاسخ و ثبت گفتگو
    |
    `--> پاسخ قابل اتکا پیدا نشد
            |
            v
        ساخت payload تشخیصی
            |
            v
        ساخت پرونده داخلی و تیکت اختیاری سهند/Jira
            |
            v
        نمایش رسید و مرحله پیگیری
```

قبل از actionهای مرحله‌ای، snapshot گفتگو ذخیره می‌شود تا کاربر بتواند به مرحله
قبل برگردد.

## مالکیت داده

| داده                 | مالک                                                       |
| -------------------- | ---------------------------------------------------------- |
| FAQ                  | بک‌اند و `server/src/faqs`                                 |
| گفتگوها              | بک‌اند و `server/src/conversations`                        |
| پرونده‌های تشخیصی    | بک‌اند و `server/src/diagnostics`                          |
| سرویس‌های قابل تعریف | بک‌اند و `server/src/services`                             |
| درختواره عیب‌یابی    | API بک‌اند؛ fallback فایل `src/assets` و graph در ArangoDB |
| تنظیمات تم           | storage مرورگر برای هر کاربر                               |

## ذخیره‌سازی و ArangoDB

در اجرای پیش‌فرض، repositoryهای بک‌اند از LowDB استفاده می‌کنند تا برنامه بدون
نیاز به سرویس خارجی قابل اجرا باشد. ArangoDB به عنوان مسیر دیتابیس عملیاتی و
graph-ready اضافه شده است و با این متغیر فعال می‌شود:

```text
DB_PROVIDER=arango
```

در حالت ArangoDB، `server/src/database/arango.ts` دیتابیس `rahyar`، collectionهای
سندی، collection لبه `troubleshooting_edges` و indexهای لازم را آماده می‌کند.
Repositoryهای عمومی FAQ، گفتگو، پرونده، تنظیمات، سرویس‌ها و کاربران در همین حالت
روی Arango کار می‌کنند. درختواره از مسیر `GET /api/troubleshooting-tree` خوانده
می‌شود؛ اگر graph خالی باشد، فایل `src/assets/troubleshooting-tree.json` seed
اولیه است.

جزئیات عملیاتی در سند [پایگاه داده ArangoDB](DATABASE_ARANGO.md) آمده است.

## اتصال تیکت

ماژول diagnostics برای هر ارجاع یک پرونده داخلی می‌سازد. اگر
`SAHAND_TICKET_URL`، شناسه service desk، شناسه request type و اطلاعات احراز هویت
سهند تنظیم شده باشند، بک‌اند پرونده را با payload رسمی سهند به endpoint خارجی
ارسال می‌کند. credentialهای سامانه مقصد هیچ‌وقت به مرورگر ارسال نمی‌شوند.

مدیر می‌تواند آدرس سرویس، هدر `Authorization`، هدر اضافی `Auth`، شناسه
پروژه/Service Desk، شناسه RequestType و نگاشت Nodeهای درختواره به RequestType را
از پنل پیکربندی تغییر دهد. این تنظیمات در LowDB ذخیره می‌شوند و نسبت به مقدارهای
env اولویت دارند. هنگام ارسال به سهند، ابتدا `treeNodeId` پرونده با نگاشت‌های
ثبت‌شده مقایسه می‌شود؛ اگر نگاشت وجود نداشته باشد، مقدار عمومی پنل و سپس env
استفاده می‌شود.

## کاتالوگ سرویس‌ها

ماژول `server/src/services` به مدیر اجازه می‌دهد سرویس‌های عملیاتی مستقل تعریف
کند. هر سرویس شامل کلید پایدار، عنوان، هدف، عنوان بخش نمایشی، متد HTTP، آدرس،
هدرهای لازم و قالب بدنه است. routeهای مدیریتی فقط برای نقش مدیر باز هستند.
کاربر فقط فهرست سرویس‌های فعال و قابل نمایش را می‌گیرد و مقدارهای محرمانه مثل
`Authorization`، `Auth`، هدرهای خام و قالب بدنه به مرورگر او ارسال نمی‌شوند.

اجرای سرویس هم از بک‌اند انجام می‌شود. بک‌اند قالب بدنه را با اطلاعات کاربر
احراز هویت‌شده مانند `username`، `fullName`، `userId`، `role` و `now` پر می‌کند
و درخواست را به سامانه مقصد می‌فرستد. صفحه کاربر براساس سرویس‌های فعال کارت
عملیاتی می‌سازد و نتیجه اجرای آخرین سرویس را در همان جریان گفتگو نشان می‌دهد.

## معماری تم

`ThemeService` تنظیمات هر کاربر را ذخیره می‌کند و CSS custom propertyهای سراسری
را روی root سند اعمال می‌کند. صفحه کاربر، ادمین، ورود و لوگو از همین tokenها
استفاده می‌کنند.

## محدودیت‌های فعلی

- درختواره هنوز از پنل ادمین قابل ویرایش نیست.
- LowDB دیتابیس همزمان و تولیدی محسوب نمی‌شود.
- SSO سازمانی پیاده‌سازی نشده است.
- رفتار نهایی سهند/Jira به مستندات API کارفرما وابسته است.
- تست end-to-end کامل هنوز لازم است.
