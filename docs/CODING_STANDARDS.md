# استاندارد کدنویسی

## هدف

کد باید قابل review، قابل تغییر و بین فرانت‌اند و بک‌اند یکدست باشد. تغییرات
کوچک، روشن و محدود به هدف مشخص بر refactorهای گسترده ترجیح دارند؛ مگر اینکه
refactor برای تحویل رفتار لازم باشد.

## قواعد عمومی

- TypeScript و Angular template checking باید strict بمانند.
- هر فایل فقط یک مسئولیت اصلی داشته باشد.
- منطق workflow دامنه داخل کامپوننت‌های نمایشی مشترک نوشته نشود.
- سرویس‌ها و مدل‌های مشترک در `core` نگهداری شوند.
- کامپوننت‌های نمایشی قابل استفاده مجدد در `shared` باشند.
- `core` نباید به featureها وابسته شود.
- مقدارهای زمان اجرا از environment یا config خوانده شوند، نه hard-code.
- رفتارهای امنیتی در بک‌اند نگه داشته شوند.

## نام‌گذاری

| مورد                   | قرارداد                         |
| ---------------------- | ------------------------------- |
| کلاس، interface و type | `PascalCase`                    |
| متغیر، تابع و property | `camelCase`                     |
| فایل و پوشه            | `kebab-case`                    |
| selectorهای Angular    | `app-kebab-case`                |
| ثابت‌ها                | نام روشن و بدون اختصار غیرضروری |

## قواعد Angular

- کامپوننت‌های standalone ترجیح دارند.
- صفحه‌های route-level باید lazy load شوند.
- کامپوننت‌های feature از `ChangeDetectionStrategy.OnPush` استفاده کنند.
- templateها ساده و قابل خواندن باشند.
- parsing، filtering و workflow به سرویس منتقل شود.
- پاسخ‌ها و payloadهای API مدل TypeScript داشته باشند.
- subscriptionها کوتاه‌عمر یا تحت مدیریت Angular باشند.
- label، focus state و دسترسی keyboard برای عناصر تعاملی رعایت شود.

## قواعد UI و SCSS

- قبل از رنگ محلی، از tokenهای سراسری تم استفاده شود.
- کنترل‌های ثابت، ابعاد پایدار داشته باشند.
- متن در desktop و mobile روی هم نیفتد.
- رفتار چیدمان در layoutهای حساس صریح باشد.
- animation با reduced motion سازگار باشد.
- الگوهای طراحی موجود بر ظاهرهای تک‌موردی ترجیح دارند.

## قواعد بک‌اند

- body درخواست‌ها با `zod` اعتبارسنجی شود.
- route moduleها کوچک و دامنه‌محور بمانند.
- مسیرهای محافظت‌شده `requireAuth` داشته باشند.
- کنترل نقش نزدیک تعریف route باشد.
- برای خطاها code پایدار برگردانده شود.
- stack trace یا secret در پاسخ API نمایش داده نشود.
- integration خارجی پشت سرویس مستقل قرار گیرد.

## مدیریت خطا

- پیام کاربر بگوید چه اتفاقی افتاده و اقدام بعدی چیست.
- جزئیات فنی در log باشند، نه UI.
- status codeها معنادار باشند.
- خطای validation نباید داده را نیمه‌کاره تغییر دهد.
- شکست اتصال تیکت نباید پرونده داخلی را از بین ببرد.

## قواعد مستندات

- تغییر رفتار باید همراه با به‌روزرسانی مستندات باشد.
- فرمان‌های setup قابل copy/paste باشند.
- مثال‌های API با schema routeها هم‌خوان باشند.
- متن مستندات فارسی، روشن و قابل فهم باشد.
- عبارت‌های انگلیسی فقط برای نام ابزار، endpoint، type و command استفاده شوند.

## کنترل کیفیت

قبل از تحویل اجرا کنید:

```powershell
npm run format:check
npm run build:all
```

در صورت دسترسی به registry:

```powershell
npm audit --omit=dev
```

قالب‌بندی پروژه با Prettier تعریف شده است؛ فرمت دستی ملاک نیست.
